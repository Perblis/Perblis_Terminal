# Terminal v2 — Decision Log

Running log of founder-approved decisions. Each entry is binding on downstream documents (FSD, TSD, design system) unless explicitly superseded here.

| # | Date | Decision | Detail |
|---|---|---|---|
| D-001 | 2026-06-11 | **Naming: Supplier / Hirer** | Replaces Owner/Renter platform-wide. "Hire" is the product word for a booking. See `01_Product_Definition.md` §1. |
| D-002 | 2026-06-11 | **Service fee = Model A: per-class rates + minimum-fee floor** | Keep the FSD per-class/per-duration rate table (Plant & Machinery 12/10/8 · Trucks & Haulage 11/9/7 · Warehousing, Terminals, Land 10/8/6). Replace the flat-fee-below-₦25,000 rule with: `service_fee = max(rate × hire_value, ₦2,500)`. Eliminates both fee cliffs (the ₦24,999→₦25,000 jump and the reverse cliff where the flat fee exceeded low monthly percentages). Display as "Service fee: 12% (min ₦2,500)". |
| D-003 | 2026-06-11 | **Big-ticket fee taper deferred to Phase 2** | Marginal rate bands / fee cap for high-value hires (anti-leakage economics) to be decided when real payments launch and big-ticket behaviour is observable. Pre-committed as a Phase 2 agenda item, not an open question. |
| D-004 | 2026-06-11 | **No minimum hire value** | The Policy Bible's ₦25,000 minimum hire value is NOT adopted. The ₦2,500 fee floor keeps small hires profitable; long-tail supply (generators, small storage) stays on-platform. |
| D-005 | 2026-06-11 | **Split fees and hirer-side fees rejected for MVP** | Hirer pays the listed price only (supplier bears the service fee), consistent with Policy Bible §4.2. Revisit only at Phase 2 alongside real payment protection. |

| D-006 | 2026-06-11 | **MVP payments: collect-only via Paystack** | Real checkout, 4h payment window, webhook-driven On Hire transition, refunds via API. Supplier payouts manual via Ops Console payout queue (weekly). Deposits deferred to Phase 2. "Simulate integrations, never simulate trust" — OTP real (Termii), ID verification real-but-manual. |
| D-007 | 2026-06-11 | **Launch: both Lagos corridors, class-anchored** | Lekki–Ajah/Mainland anchored by Plant & Machinery + Trucks & Haulage; Apapa–Tin Can anchored by Terminals/Warehousing/Land. Per-corridor supply targets in 03 §8. (Founder chose dual-corridor over single-corridor recommendation.) |
| D-008 | 2026-06-11 | **Duration pricing: best-price rule** | `hire_value = min(daily×d, weekly×ceil(d/7), monthly×ceil(d/30))` over supplier-set rates; winning scheme sets the service-fee tier. Replaces fixed boundary bands; kills the "8 days = 2 weekly" unfairness. |

| D-009 | 2026-06-11 | **API framework: DRF 3.17** (overrides locked Django Ninja) | Decisive: rest_framework_gis for the map-first core; multi-maintainer ecosystem; agent code accuracy. Stack: Django 6.0.x + DRF + simplejwt + drf-spectacular + rest_framework_gis. |
| D-010 | 2026-06-11 | **Tasks: Django 6 native Tasks + django-tasks** (overrides locked Celery+Redis) | Postgres as broker; no Redis anywhere in the stack. Celery reconsidered post-MVP only for heavy fan-out. |
| D-011 | 2026-06-11 | **Realtime: Ably free tier** (overrides locked Pusher) | 200 connections / 6M msgs free; $29 paid cliff vs Pusher's $49 at 100 connections. |
| D-012 | 2026-06-11 | **Hosting: Railway Hobby (~$10–15/mo)** + Cloudflare Workers for the Supplier Portal (Vercel Hobby forbids commercial use) + R2 media + local-Android/EAS-free-iOS builds | Fits the $25/mo cap with ~$10 headroom. Usage cap to be set on Railway. |
| D-013 | 2026-06-11 | **Maps: MapLibre GL + OpenFreeMap tiles + LocationIQ geocoding** (pin-drop-first UX) | Decided by ToS, not price: Mapbox free tier forbids storing temporary-geocoding results (our core write path) and Vercel-style traps. Protomaps-on-R2 documented as tile fallback. Stands per 04 §5 unless founder objects. |

| D-014 | 2026-06-11 | **Service fee is supplier-confidential** | Hirers see only the total they pay — no service-fee or payout figures anywhere on hirer surfaces (request preview, hire detail, receipt, emails). Fee breakdown (Hire Value / Service Fee / You Receive) renders on supplier and Ops surfaces only. ToS discloses the fee model in legal terms. Refines D-005. |

| D-015 | 2026-06-11 | **Cancellation withholdings ARE payouts (FSD §7.6 wins over §3.2)** | Resolves the conflict between FSD §3.2 ("a payout is never created for a hire that did not complete") and §7.6 (hirer cancels ≤72h before start → the withheld day "becomes a supplier payout (due)"). The §7.6 rule stands: a cancellation-withholding payout is the one exception to the completion rule — the supplier is compensated for late cancellation. §3.2 amended to carry the exception. |
| D-016 | 2026-06-11 | **Handover capture works offline (ux/02 S11 wins over TSD §6)** | Resolves the conflict between TSD §6 ("mutations require connectivity") and ux/02 S11 (offline capture, photos queue, record submits on reconnect). Handover is the one allowed offline mutation — it happens in the field at sites with poor signal, and a lost capture damages the dispute-evidence story. TSD §6 amended. |

| D-017 | 2026-06-15 | **MVP payment provider = Bachs.io (collect-only), superseding Paystack in D-006** | Provider changes; the rest of D-006 stands — collect-only, 4h payment window, webhook-driven On-Hire transition, refunds via API, **manual weekly payouts via the Ops queue** (Bachs Payouts API is Limited-Access and pays only to our own destinations — no split/escrow, so manual payout is retained, not a regression). **Integration contract (Wave 4):** base `https://api.bachs.io/v1` / sandbox `https://sandbox-api.bachs.io/v1`; auth `Authorization: Bearer sk_{live,sandbox}_…`; collect via `POST /v1/checkouts` (ad-hoc NGN); **verify-before-transition** via `GET /v1/payments/charges/{charge_id}` (require `status=SUCCEEDED` + matching amount + currency); webhook `collection.succeeded` signed **HMAC-SHA256 hex** over `"{X-Bachs-Timestamp}.{raw_body}"` with a **per-endpoint** secret (≠ API key), 5-min tolerance, dedup on envelope `id` (at-least-once delivery). **Money boundary:** Bachs uses decimal-naira strings (`"75000.00"`); domain stays integer kobo (commandment 2) — convert only inside the Bachs adapter. No monthly fee (per-transaction only) → within the $25/mo cap (D-012). Supersedes the "via Paystack" provider choice in D-006 and the Paystack references in design.md §3/§9 and TSD §6 (amend when Wave 4 builds payments). **Confirm with Bachs before Wave 4 launch:** marketplace settlement model (collect-and-payout vs Connected Accounts), Payouts Limited-Access enablement, exact NGN bank-transfer fee, production rate limit, refund-completion webhook event name, and the one-refund-per-charge constraint vs our refund model. Founder noted Bachs is early-stage (some Beta/"coming-soon" features) — accepted. |

| D-018 | 2026-06-19 | **MVP payment provider switched back to Paystack (collect-only), superseding D-017's Bachs.io** | Founder call: Paystack is the active MVP provider. The rest of D-006/D-017 stands — collect-only, 4h payment window, webhook-driven On-Hire transition, refunds via API, manual weekly payouts via the Ops queue. **Provider is now pluggable** behind `payments.gateway` (`settings.PAYMENT_PROVIDER = paystack \| bachs`); the Bachs adapter is retained, not deleted, so the choice can flip again without code change. **Paystack integration contract:** base `https://api.paystack.co`; auth `Authorization: Bearer sk_{test,live}_…`; collect via `POST /transaction/initialize` (amount in **integer kobo**, `email` required, our `reference`); **verify-before-transition** via `GET /transaction/verify/{reference}` (require `data.status=success` + matching amount + currency); webhook `charge.success` signed **HMAC-SHA512 hex of the raw body with the secret key** (header `x-paystack-signature`, no timestamp), deduped on `(event, reference)`. **Money boundary:** Paystack speaks integer kobo natively — no decimal conversion (simpler than Bachs). Supersedes D-017's provider choice; the kobo/state-machine/§7.6/D-014 domain is unchanged. |

| D-019 | 2026-06-22 | **Supplier Portal UI: bespoke token-driven components (overrides "shadcn/ui restyled" in TSD §5 / CLAUDE.md / wave-7.md)** | Founder call at the Wave 7 start: hand-build the portal component library directly from the design-system chapters (05/06/07), styled **only** from `@terminal/tokens` (no literal hex/px — design.md §5), rather than restyling shadcn/ui. Rationale: maximum fidelity to the "Heavy Duty" language (manifest blocks, stamps, hazard stripes, the supplier/hirer `LockedTerms` split, density-aware DataTable, the CSS-grid CalendarGantt) and a visually distinctive result. Headless a11y primitives (focus-trap, listbox) are hand-rolled by default; an individual unstyled `@radix-ui/react-*` may be pulled in only where a specific behaviour is too costly to hand-build. **No contract impact** (presentation only; frozen API contracts untouched). Supersedes the "shadcn/ui" wording in TSD §5, CLAUDE.md (stack), and `docs/waves/wave-7.md` §7A — amend those when convenient; this entry is binding. |

| D-020 | 2026-07-03 | **Portal headless-primitive allowlist (refines D-019)** | Ratified with the Wave 7 build-plan approval. D-019's "hand-roll by default" stands; the following **unstyled/headless** packages are pre-approved where hand-building the behaviour is disproportionate: `@radix-ui/react-dialog` (focus-trap + scroll-lock + aria for modals), `@radix-ui/react-popover`, `@radix-ui/react-tooltip`, `@radix-ui/react-slot` (composition primitive), `@tanstack/react-table` (headless table state for the P3/P6 DataTables), `@tanstack/react-virtual` (row virtualisation for large fleets). All rendering/styling stays bespoke from `@terminal/tokens` (design.md §5); anything outside this list still needs a new decision. No contract impact. |

| D-021 | 2026-07-03 | **Corner-bracket panel motif (extends design-system 01 §2 M4)** | Founder vision call (wave-7-vision.md): the thin L-shaped corner registration marks — previously **hero imagery only** — may additionally frame **key money/trust panels** in the portal: dashboard stat cards, the P7 LockedTerms panel, and the P11 bank-details vault. Rendered as 1.5px, 12px-arm marks in `ink-200` on light fields (`ink-900 @ 60%` on photography, unchanged). The one-motif-per-screen restraint rule (01 §2) still binds — a screen using corner-bracket panels uses no hazard stripe/stamp/manifest motif except where 01 §2 already allows pairing (receipts). Amends 01 §2 M4's "hero imagery only" scope; the design-system chapters otherwise stand. |
